# 08 — Simulation Pack

> **Series:** BoVerse Workflow Creator Framework
> **Document:** 08 of the corpus — the validation loop and the demo artifact
> **Status:** Portable IP / handoff artifact. Implementation-agnostic. An engineer on any stack must be able to implement against this document without further questions.
> **Cross-references:** The 12 discovery categories are defined in [01 — Workflow Discovery Framework](01-discovery-framework.md). The 9 archetypes are defined in [02 — Workflow Archetype Framework](02-archetype-framework.md). The canonical tables (§9.1–§9.14) this document reads from and writes synthetic instances of are defined in [04 — Canonical Workflow Design Schema](04-canonical-schema.md). The Simulation Pack is produced by the Discovery Swarm (Swarm 1) alongside the Workflow Design Specification and is consumed as a demo/sales artifact and as the input to human confirmation before Build (Swarm 2).

## 1. Purpose of this document

The Simulation Pack is the artifact that proves Discovery actually understood the workflow **before a single BoVerse object is built**. It is generated by the Discovery Swarm directly from the canonical store (the §9.x tables in [04](04-canonical-schema.md)), and it does its work by running a deliberate inversion:

> **Generate a sample OUTPUT first; then work BACKWARD to generate the sample INPUTS that would have to exist to produce it.**

This direction is the whole point. It is far easier to write plausible inputs that lead to a vague output than it is to commit to a concrete output and then be forced to admit exactly what you needed to produce it. By committing to the output first, the system exposes its own gaps.

The pack is therefore two things at once:

1. **A validation loop** — an internal self-test of Discovery's comprehension. Each artifact in the pack can *fail*, and each kind of failure points at a specific defect in the canonical store.
2. **A demo and sales tool** — a single screen-ready narrative that tells a prospective client: *here is what we think you want, here is what it would produce, here is what we need from you, here is what we can infer for you, and here is what is still uncertain.*

The same artifact serves both. A pack good enough to demo is a pack good enough to validate, because both audiences are asking the same question: *does this system actually understand my work?*

### 1.1 Where determinism lives in the pack

Generating a *plausible* sample output from a partly-known workflow is **probabilistic** — it is LLM-driven, exactly like the extraction step in [01 §2.2](01-discovery-framework.md) and [04 §1](04-canonical-schema.md). The same canonical store run twice may yield slightly different sample numbers, wording, or example records.

> **Determinism does not come from the generation step. It comes from the contract the generated artifacts must satisfy.**

- The **probabilistic part** is: "given this canonical store, imagine a realistic run and its artifacts."
- The **deterministic constraints** are: (a) every sample artifact must conform to the **canonical schema** — a sample output must match its Output row (§9.3), a sample input must match an Input row of type `required_workflow_input` or `both` (§9.4), a sample rule firing must match a Rule row (§9.11); and (b) every claim the pack makes about *why* a value appears must resolve to a **provenance** row (§9.14) or to a row in the **Missing Information / Ambiguity Ledger** (§9.12).

So the imagined run is creative; the *frame it must fit* is fixed. Anything the generator cannot ground in a canonical row is not allowed to appear as fact in the pack — it appears as an **inferred value** (clearly labeled, defaulted by the rules layer) or as an **open question** (a ledger row). Nothing is silently invented. This is the same "LLM proposes, schema disposes" discipline used everywhere in the corpus, applied to synthesis instead of extraction.

## 2. The validation loop

The pack is generated as an ordered loop. Each stage produces an artifact, and each stage has a **failure mode** that, when triggered, writes back into the canonical store (almost always as a new or updated row in the ledger, §9.12) rather than producing a broken pack.

| Order | The system tries to… | If it cannot… | What that failure proves | Write-back |
|---|---|---|---|---|
| 1 | Generate a plausible **sample output** conforming to the Output row(s) (§9.3) | …it does not actually understand the workflow's purpose | Outcome/Output (§9.2/§9.3) is too thin to be buildable | Critical gap in ledger (§9.12); block the spec |
| 2 | Work **backward** to the **sample inputs** required to produce that output | …a required input has no row, or a row has no realistic shape | A required input is missing or under-specified (§9.4) | Gap in ledger; possibly a clarifying question |
| 3 | Populate the **canonical table rows** and **extracted facts** the run would touch | …a field has no value and no rule can default it | The extraction left a hole the rules layer cannot fill | Gap in ledger; mark affected field low-confidence |
| 4 | Walk the **intermediate steps** (§9.8) end to end with the sample data | …a step cannot consume its declared input or produce its output | The process model (§9.8) has a break or a missing rule (§9.11) | Gap in ledger tied to `affected_step` |
| 5 | Surface every **assumption** the synthetic run had to make | …an assumption is load-bearing and unconfirmed | The pack is resting on guesses a human must ratify | Ledger row, `resolution_status = assumed` |

Read the failure column as a diagnostic legend. The pack is valuable precisely because it can break:

- **If the system cannot generate a plausible output, it does not understand the workflow.** (Stage 1)
- **The sample output exposes missing inputs.** Committing to a concrete deliverable forces every input it depends on into the open. (Stage 2)
- **The sample input pack exposes unrealistic assumptions.** Writing a *realistic* input record reveals where the schema assumed data that the client's systems do not actually carry. (Stages 2–3)
- **The missing-info ledger exposes open questions.** Everything stages 1–4 could not ground becomes an explicit, ranked question rather than a silent default. (Stage 5)

A pack that passes all five stages with no critical ledger rows is the green light to hand the Workflow Design Specification to Build (Swarm 2). A pack that accumulates critical, blocking ledger rows is the system telling its operators *Discovery is not done* — and telling the client, in the same breath, exactly what it still needs.

### 2.1 Regeneration is cheap and versioned

Because the pack is a pure function of the canonical store plus the rules layer, it is **regenerated** whenever the store changes (a clarifying question is answered, a fact is corrected). Every artifact in the pack carries the `version` of the facts it was built from (§9.14 `version`), so a client can watch the pack sharpen as questions are answered: a value that was an *inferred* guess in v1 becomes a *grounded* fact in v2 once the supporting evidence arrives. The demo narrative (§5) is designed to make this sharpening visible and persuasive.

## 3. Pack contents

The pack is a set of named artifacts. Each is marked **Mandatory** (always present in every pack, for every archetype) or **Archetype-specific** (present only when the workflow's archetype, per [02](02-archetype-framework.md), calls for it). Every artifact resolves to one or more canonical tables; the **Source rows** column names them by their §9.x anchor in [04](04-canonical-schema.md).

| # | Artifact | Mandatory / Archetype-specific | What it is | Source rows (CANON §9.x) |
|---|---|---|---|---|
| A1 | **Sample user intake** | Mandatory | A realistic reconstruction of the request a user would type/upload to start a run — the synthetic equivalent of the freeform ask. | Workflow Identity §9.1 (`stated_problem`); Trigger §9.7; Input/Evidence §9.4 (`required_workflow_input`) |
| A2 | **Sample uploaded file(s)** | Mandatory | One or more synthetic input documents the run would receive (a spreadsheet, a PDF, a record export), shaped to the declared input format. | Input/Evidence §9.4; Systems/Connector §9.9 (`source_system`, `data_objects_accessed`) |
| A3 | **Sample canonical table rows** | Mandatory | The actual §9.x rows the run would populate, rendered as data — proof the run lands in the schema. | All touched tables; minimally Workflow Identity §9.1, Output §9.3, Input §9.4, Process/Step §9.8 |
| A4 | **Sample extracted facts** | Mandatory | The fact-level view: each value with its confidence and provenance, distinguishing **grounded** facts from **inferred/defaulted** ones. | Audit/Provenance §9.14 (`fact_id`, `extracted_value`, `confidence_score`, `source_document`) |
| A5 | **Sample missing-question ledger** | Mandatory | The open questions this run would raise — the demo's "here is what we need from you / what is still uncertain." | Missing Information / Ambiguity Ledger §9.12 |
| A6 | **Sample intermediate workflow steps** | Mandatory | A trace of the run walking the ordered steps, showing input→step→output at each stage, with the determinism flag per step. | Process/Step §9.8; Rule/Decision §9.11 |
| A7 | **Sample final output** | Mandatory | The committed deliverable, generated first (stage 1), conforming to the Output row's `output_type`, `output_format`, and `required_sections`. | Output §9.3 |
| A8 | **Sample dashboard** | Archetype-specific | A view across runs (metrics, queues, status). Present for **App**, **Decision-Support App**, **Integrated Workflow**, **Bridge**, **Intelligence Layer**, **Operating Layer / OSO**. Omitted for **Workflow Component**, usually for **Mini-App** and **Sharp Point Solution**. | Output §9.3 (`output_type = dashboard`); Outcome §9.2 (`success_metric`) |
| A9 | **Sample exception case** | Archetype-specific (Mandatory wherever any Exception is recorded) | A second run that deliberately trips a known failure mode and shows the handling path. Present whenever §9.8 `error_conditions` or any exception handling exists; effectively mandatory for **Integrated Workflow**, **Bridge**, and any workflow with HITL. | Process/Step §9.8 (`error_conditions`); HITL §9.6 (`rejection_path`, `escalation_path`); Rule §9.11 (`exception_handling`) |
| A10 | **Sample approval flow** | Archetype-specific | A walkthrough of a human review/approval gate: what the reviewer sees, the decision, and the downstream effect. Present whenever a HITL row exists (§9.6) — common for **Sharp Point**, **App**, **Decision-Support**, **Integrated Workflow**. Absent when the happy path has no human (e.g. a pure **Workflow Component** or a no-human **Bridge**). | Human-in-the-Loop §9.6; Actor/Role §9.5; Output §9.3 (`approval_required`) |
| A11 | **Sample audit log** | Mandatory | The per-run trail: which inputs were used, which rules fired, which facts were extracted, who reviewed, with timestamps and versions. | Audit/Provenance §9.14; HITL §9.6 (`audit_required`) |

> **Why A11 is mandatory even when the Audit category is light.** [01 §5.11](01-discovery-framework.md) treats audit as first-class. Even a workflow with no regulatory requirement produces *some* trail when it runs; the pack must show at least the minimal trail (inputs used, rules fired, output version) so the client sees the system is explainable. The depth of the trail scales with the workflow's audit requirement, but its presence does not.

### 3.1 Mandatory-vs-archetype quick map

A compact lookup for which optional artifacts (A8 dashboard, A9 exception, A10 approval) a generator should expect to produce, by primary archetype ([02 §3](02-archetype-framework.md)). "—" means *omit by default*; "cond." means *include only if the relevant canonical rows exist*.

| Archetype | A8 Dashboard | A9 Exception | A10 Approval |
|---|---|---|---|
| Workflow Component | — | cond. (if `error_conditions`) | — |
| Mini-App | — | cond. | cond. (if HITL) |
| Sharp Point Solution | cond. | cond. | cond. (common) |
| Bridge | yes (status board) | yes | cond. (sign-off bridges) |
| App | yes | yes | cond. |
| Decision-Support App | yes | cond. | cond. (often, for accepting a recommendation) |
| Integrated Workflow | yes | yes | yes |
| Intelligence Layer | yes (query/coverage view) | cond. | — |
| Operating Layer / OSO | yes | yes | yes |

The A1–A7 and A11 artifacts are mandatory across every row above. Only A8/A9/A10 vary, and they vary as a deterministic function of the archetype plus the presence of the source rows — never as a free choice of the generator.

### 3.2 Generation order vs. presentation order

The artifacts are **generated** in the validation-loop order of §2 (output first, then backward to inputs, then forward through steps). They are **presented** in the narrative order of §5 (intake → output → inputs → inferences → uncertainties). Implementations must not confuse the two: the backward generation is what makes the pack honest; the forward presentation is what makes it persuasive.

## 4. Per-artifact specification

Each artifact below states its **goal**, its **canonical binding** (which §9.x fields it draws from and must conform to), and its **failure mode** (the validation signal it raises). Field names are taken verbatim from [04](04-canonical-schema.md).

### A1 — Sample user intake *(Mandatory)*

- **Goal.** Reconstruct the messy ask a real user would submit, so the demo opens where the client lives: a sentence or two of intent plus "here's what I've got."
- **Canonical binding.** Mirrors Workflow Identity §9.1 `stated_problem`; lists the inputs the user would attach, drawn from Input/Evidence §9.4 rows with `input_type ∈ {required_workflow_input, both}`; names the Trigger §9.7 (`trigger_type`, `trigger_description`) that this intake represents.
- **Failure mode.** If the generator cannot write an intake that plausibly leads to the committed output (A7), the Outcome/Output framing is too thin → critical ledger row (§9.12), block.

### A2 — Sample uploaded file(s) *(Mandatory)*

- **Goal.** Produce the synthetic raw input(s) — a spreadsheet with rows, a PDF with sections, a JSON export — in the declared format, realistic enough that the client recognizes their own data.
- **Canonical binding.** One file per relevant Input/Evidence §9.4 row, conforming to that row's `format` and `structured_or_unstructured`; where the input comes from a system, the file's shape reflects Systems/Connector §9.9 `data_objects_accessed`.
- **Failure mode.** If a required input cannot be given a realistic shape (e.g. "labor rate table" with no idea what columns it has), that under-specification is a gap → ledger row, often a clarifying question.

### A3 — Sample canonical table rows *(Mandatory)*

- **Goal.** Show the schema *populated*: the literal §9.x rows this one run produces, rendered as a small set of tables. This is the artifact that proves the messy intake (A1) and files (A2) actually land in the canonical store.
- **Canonical binding.** Real rows for every table the run touches, minimally Workflow Identity §9.1, Output §9.3, the relevant Input §9.4 rows, and the Process/Step §9.8 rows. Each row carries its `confidence_score`.
- **Failure mode.** A required field with no value and no rule (§9.11) able to default it → the extraction left a hole → ledger row; mark the field low-confidence.

### A4 — Sample extracted facts *(Mandatory)*

- **Goal.** The fact-level, provenance-first view. For each value that mattered, show: the value, its `confidence_score`, where it came from, and whether it is **grounded** (traces to a real evidence document) or **inferred/defaulted** (supplied by the rules layer). This is the artifact that earns trust.
- **Canonical binding.** One entry per Audit/Provenance §9.14 row: `fact_id`, `extracted_value`, `extraction_method`, `confidence_score`, `source_document`, `source_location`, `review_status`. Grounded vs inferred is read from `extraction_method` (`llm_extraction`/`direct_field`/`ocr`/`table_parse` = grounded in evidence; `rule_derived` = inferred/defaulted).
- **Failure mode.** A value with no provenance row and no rule derivation is illegal in the pack; it must be demoted to an open question (§9.12) before the pack renders.

### A5 — Sample missing-question ledger *(Mandatory)*

- **Goal.** The honest list: "here is what we still need from you, and here is what we assumed in the meantime." In the demo this is reframed positively ("a few quick questions will lock this in"), but it is the same data.
- **Canonical binding.** Direct projection of the Missing Information / Ambiguity Ledger §9.12: `missing_attribute`, `why_it_matters`, `suggested_question`, `severity`, `blocking_status`, `resolution_status`, with links to `affected_output` and `affected_step`. The pack surfaces only `severity ∈ {critical, high}` questions to the client by default (consistent with the question policy: only high-value gaps earn a human's time); lower-severity gaps are retained but collapsed.
- **Failure mode.** This artifact does not fail — it *is* the accumulation of every other artifact's failures. An empty critical/high ledger is the pass condition for the whole pack.

### A6 — Sample intermediate workflow steps *(Mandatory)*

- **Goal.** Trace the run through the process so the client sees *how* the output was produced, step by step, and where the system reasons vs. follows a rule vs. asks a human.
- **Canonical binding.** Ordered by Process/Step §9.8 `sequence_order`; for each step show `step_name`, the input it consumed (`input_required`), the output it produced (`output_produced`), and the three determinism flags (`deterministic_rule_available`, `probabilistic_reasoning_required`, `hitl_required`). Where a deterministic rule fires, name the Rule §9.11 row (`rule_name`, `condition`, `action`).
- **Failure mode.** A step that cannot consume its declared input or produce its declared output → a break in the process model → ledger row tied to `affected_step`.

### A7 — Sample final output *(Mandatory, generated FIRST)*

- **Goal.** The committed deliverable. This is generated **before** any input (stage 1 of §2) precisely so its existence forces the inputs into the open. It is the single most important artifact for both validation and demo.
- **Canonical binding.** Conforms exactly to the Output §9.3 row: `output_type`, `output_format`, `required_sections`, `required_fields`. If a sample output existed in the evidence (§9.3 `sample_output_available = true`), the generated output must be consistent with it.
- **Failure mode.** Inability to produce a conforming output is the top-level validation failure (stage 1): Discovery does not understand the workflow → block the spec with a critical ledger row.

### A8 — Sample dashboard *(Archetype-specific)*

- **Goal.** Show value *across* runs — queues, throughput, status, coverage — for archetypes where the product is an ongoing surface, not a one-shot artifact.
- **Canonical binding.** Present when an Output §9.3 row has `output_type = dashboard`, or when the archetype implies a standing surface (§3.1 map). Metrics are drawn from Outcome §9.2 `success_metric` and the workflow's own run records.
- **Failure mode.** If the archetype calls for a dashboard but no `success_metric` exists to populate it, that is a gap in the success framing → ledger row (low severity unless the dashboard is the primary deliverable).

### A9 — Sample exception case *(Archetype-specific; Mandatory where exceptions exist)*

- **Goal.** Prove the workflow survives contact with reality by running a second, deliberately-broken case and showing the handling path. This is the artifact that separates a demo-only build from a production-ready one ([01 §5.10](01-discovery-framework.md)).
- **Canonical binding.** Driven by Process/Step §9.8 `error_conditions` and Rule §9.11 `exception_handling`; the handling path resolves to HITL §9.6 (`rejection_path`, `escalation_path`) or to a rule's fallback. Present whenever any such row exists.
- **Failure mode.** A named `error_condition` with no defined handling (no rule, no HITL, no fallback) → ledger row: the workflow has an unhandled failure mode.

### A10 — Sample approval flow *(Archetype-specific)*

- **Goal.** Walk a human review gate end to end: what the reviewer sees, what they decide, and what happens next on approve / reject / escalate.
- **Canonical binding.** One per Human-in-the-Loop §9.6 row: `human_role` (which **must** resolve to an Actor/Role §9.5 `role_name`, per the HITL→Actor invariant in [04 §2](04-canonical-schema.md)), `review_trigger`, `confidence_threshold`, `required_action`, `rejection_path`, `escalation_path`. The reviewer's view lists `evidence_required`.
- **Failure mode.** A HITL row whose `human_role` does not resolve to an actor is an invariant violation → the pack must not render it; raise it for correction.

### A11 — Sample audit log *(Mandatory)*

- **Goal.** The explainability trail for the one sample run: a chronological record a human (or auditor) could use to answer "why did it produce this?"
- **Canonical binding.** Composed from Audit/Provenance §9.14 rows for the run (each `fact_id` with `source_document`, `extraction_method`, `timestamp`, `version`, `review_status`), plus the rules that fired (§9.11) and the human actions taken (§9.6 where `audit_required = true`).
- **Failure mode.** Does not block the pack; a thin trail is acceptable for low-audit workflows, but the trail must exist. Absence of any trail for a workflow whose Outcome cited compliance/risk (§9.2 `risk_reduction`) is itself a gap.

## 5. The demo narrative

For the client-facing demo, the pack is presented as a five-beat story. Each beat maps to specific artifacts and to a specific reassurance. This is the order in §3.2's "presentation" column — note it is *not* the generation order.

| Beat | The claim to the client | Backed by artifacts | Backed by canonical rows |
|---|---|---|---|
| 1. **What we think you want** | "Here's the job, in your words, as we understood it." | A1 intake; the framing | Workflow Identity §9.1 (`stated_problem`, `inferred_problem`, `primary_objective`); Outcome §9.2 |
| 2. **What it would produce** | "Here's the actual thing you'd get." | A7 final output; A8 dashboard | Output §9.3 |
| 3. **What we need from you** | "Here's what you'd feed it each run." | A1 + A2 inputs | Input/Evidence §9.4 (`required_workflow_input`/`both`) |
| 4. **What we can infer for you** | "Here's what you *don't* have to supply — we derive it." | A4 facts marked inferred/defaulted; A6 rule firings | Rule/Decision §9.11; Knowledge Source §9.10; A4 entries with `extraction_method = rule_derived` |
| 5. **What's still uncertain** | "Here are a few questions to lock it in." | A5 ledger (critical/high only) | Missing Information / Ambiguity Ledger §9.12 |

Beat 4 is the differentiator and deserves emphasis in any demo: it is where the system shows it is not a form the client fills out but an engine that *reasons on their behalf* — every inferred value is a thing they did not have to think about, and every one is auditable back to a rule or a piece of evidence (A4/A11). Beat 5 turns the system's gaps into a short, respectful ask rather than a wall of unknowns, which is exactly why only `critical`/`high` ledger rows surface here.

### 6.2 Bridge flow — Bridge

**Archetype.** Bridge — coordinating status/handoffs across multiple existing tools; the value is *flow*, not a single artifact. Per the §3.1 map: A8 status board *yes*, A9 exception *yes*, A10 approval *cond.* Note that the "final output" (A7) for a Bridge is **a state change / routing action**, not a document — consistent with [01 §5.2](01-discovery-framework.md) (`output_type = action`/`state_change`).

**A1 — Sample user intake.** *(→ §9.1, §9.7, §9.4)*
> "Design work keeps stalling. A Figma comment asks for a change, someone mentions it in Slack, but no Asana task gets made and it falls through the cracks. I want comments that ask for changes to automatically become tracked tasks, and I want to see status in one place."
> Trigger §9.7: `trigger_type = event` (a new Figma comment / Slack message). Named systems §9.9: **Slack**, **Figma**, **Asana**.

**A2 — Sample uploaded / connected inputs.** *(→ §9.4, §9.9)*
Bridges ingest *live system objects*, not uploaded files. The synthetic inputs are sample records from each connector:

| Source system (§9.9) | `data_objects_accessed` | Sample object (synthetic) |
|---|---|---|
| Figma | `Comment` | "Can we bump the hero CTA to 18px and make it green?" on frame *Home/Hero*. |
| Slack | `Message` | Thread in `#design`: "saw the Figma comment on the hero, can someone action it?" |
| Asana | `Task` | *(none yet — this is the gap the bridge closes)* |

**A7 — Sample final output (state change, generated FIRST).** *(→ §9.3)*
Output §9.3: `output_type = action` + `state_change`. The committed deliverable is: **an Asana task exists, linked to the Figma comment and the Slack thread, with the right assignee and status.**

| Field | Value (synthetic) |
|---|---|
| Asana task title | "Hero CTA: 18px, green (from Figma comment)" |
| Linked back to | Figma comment URL + Slack thread URL |
| Assignee | Designer who owns *Home/Hero* |
| Status | `Open` → moves to `In Review` when Figma comment resolves |

**Backward step (stage 2).** Committing to "task has the right assignee" forces an input the intake never mentioned: *a mapping from Figma frame → owning designer*. That requirement surfaces only because the output demanded it → ledger row (§9.12).

**A6 — Sample steps (routing trace).** *(→ §9.8, §9.11)*

| Order | Step | In → Out | Determinism flags | Rule (§9.11) |
|---|---|---|---|---|
| 1 | Classify comment intent | Figma comment → {change-request? y/n} | reasoning=true | — |
| 2 | Correlate Slack mention | comment + thread → linked context | reasoning=true | — |
| 3 | Resolve assignee | frame → owning designer | rule=true | R-2 frame→owner map |
| 4 | Create/Update Asana task | context → task (create or update) | rule=true | R-3 "create vs update by link key" |
| 5 | Sync status back | Asana status ↔ Figma resolution | rule=true | R-4 status map |

**A8 — Sample dashboard (status board).** *(→ §9.3, §9.2)*
The Bridge's primary surface. Synthetic board:

| Item | Figma | Slack | Asana | State |
|---|---|---|---|---|
| Hero CTA color/size | commented | discussed | task created | In Review |
| Footer link fix | commented | — | task created | Open |
| Nav spacing | commented | discussed | **not created** | ⚠ stalled (exception) |

`success_metric` (§9.2): "0 change-request comments without a tracked task within 1 hour."

**A9 — Sample exception case.** *(→ §9.8, §9.11, §9.6)*
The *Nav spacing* row: a Figma comment classified as a change-request (step 1) but step 3 cannot resolve an assignee (frame not in the owner map). `error_condition = "no owner mapping"`; handling = `hold_for_review` + notify the RevOps/design-ops owner (§9.6 `escalation_path`). The board shows it as stalled rather than silently dropping it — which is the exact pain the Bridge exists to kill.

**A4 — Sample extracted facts.** *(→ §9.14)*

| Fact | Value | `extraction_method` | Grounded? |
|---|---|---|---|
| Comment is a change-request | true | `llm_extraction` | Grounded (Figma comment text) |
| Linked Slack thread | `#design` ts=… | `direct_field` | Grounded (Slack API) |
| Frame owner | "designer A" | `rule_derived` | **Inferred** (R-2 map — unconfirmed for some frames) |

**A5 — Sample ledger.** *(→ §9.12)*

| `missing_attribute` | `suggested_question` | `severity` | `blocking_status` |
|---|---|---|---|
| Figma-frame → owner mapping | "Who owns each Figma frame/page, so tasks get the right assignee?" | high | blocking (for assignment) |
| Status-sync direction | "Should resolving the Figma comment close the Asana task, or only move it to In Review?" | medium | non_blocking |

**A11 — Sample audit log.** *(→ §9.14)*
Per routed item: Figma comment id, classification conf, Slack thread linked, rule R-2/R-3/R-4 firings, Asana task id created, status transitions — each with `timestamp` and `version`, so a wrong assignment can be traced to the R-2 mapping that produced it.

**Validation read.** Note what is **absent**: there is no document output (A7 is a state change), and there is no human in the happy path, so A10 (approval) is omitted — both correct for a Bridge per [02 §3.4](02-archetype-framework.md). The pack instead leans on A8 (the status board) and A9 (the stalled-item exception). One **blocking** ledger row (the owner mapping) tells the operators Discovery is not done; the demo tells the client "give us your frame-owner map and this runs itself."

### 6.3 Reconciliation — Workflow Component *(short sketch)*

To show how the pack **collapses** for the smallest archetype: a Workflow Component that reconciles a bank export against a ledger and returns the deltas ([02 §3.1](02-archetype-framework.md)).

- **Present:** A1 (the call, not a user intake — "reconcile these two tables"), A2 (two synthetic tables), A3/A4 (canonical rows + facts), A6 (match → emit-deltas steps), **A7 (the delta set — the returned object, generated first)**, A11 (a minimal trail: which rows matched on what rule).
- **Omitted by the §3.1 map:** A8 dashboard (—), A10 approval (— : no human, output is consumed by another workflow), and A9 only if `error_conditions` exist (e.g. ambiguous one-to-many match → returned as an exception in the delta set rather than a human queue).
- **The point.** Even at its smallest, the pack still generates the **output first** (the delta set) and works backward to the two inputs, and still produces an audit trail. The validation loop is identical; only the artifact count shrinks. This is the architecture-determines-build-path principle ([01 §2](01-discovery-framework.md), [02 §1](02-archetype-framework.md)) made visible in the demo: a small archetype yields a small pack, deterministically.
