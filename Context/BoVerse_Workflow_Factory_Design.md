## Problem Statement

BoVerse is an enterprise AI company building a "Workflow Factory": a meta-system that takes raw client materials (sample quotes, email threads, voice-note transcripts, spreadsheets), reasons BACKWARDS from the artifact to the embedded process, and outputs two artifacts.

1. A plain-English Word doc spec the client can read and amend.
2. A runnable simulation of the workflow against synthetic data with intentional edge cases, plus real client data when authorized.

The system serves the underserved $5M-$30M SMB segment (no IT department, ~12-50 staff, software stack = QuickBooks + Excel + email + one industry tool). Their operational expertise lives as tribal knowledge in one person's head: the estimator who has priced jobs for 15 years carries the company's entire margin logic. That knowledge is fragile, doesn't survive a resignation, and can't be applied consistently. The Workflow Factory productizes consulting BoVerse currently delivers as bespoke engagements.

This is being built as an independent study running Sep 2026 to Apr 2027 (48 contact hours, ~320-480 actual builder hours), with three graded deliverables and a final 20-minute presentation in the week of April 19-23, 2027. **Honest framing per outside voice finding #1**: the build phase is ~17 weeks (weeks 1-17), the remaining 8 weeks (18-25) are productization roadmap + Deliverable 3 writeup + final presentation prep.

## What Makes This Cool

1. **The primitive grammar is real, not aesthetic.** Five workflow primitives (ingest, transform, validate, action, feedback) form a complete vocabulary. The Donella Meadows insight applied correctly. The system builds from a fixed grammar; expertise lives in the configuration. Caveat: the universality claim is a falsification experiment, not a confirmation test (outside voice finding #3). Second vertical is THE test.

2. **The mirror moment.** The system reads four artifacts in 60 seconds and produces a plain-English summary that captures the client's business better than they could articulate. Backwards inference from artifacts extracts judgment that lives only in someone's head.

3. **The Word doc as iteration interface, with chat as a second surface.** Clients can mark up a doc OR type plain-English corrections in a chat panel. System diffs and regenerates only what changed. Two surfaces, one backend.

4. **The simulation pack does three jobs at once.** It defines the input contract (the schema real batch data must match), it provides the demo data that makes the output bundle immediately tangible, AND it becomes the regression test suite that travels with every iteration. 10 rows, 7 happy-path plus 3 deliberate edge cases.

5. **Real execution, not narrative.** The factory generates actual runnable workflow code (Claude Agent SDK subagents), not just an LLM-written story. The trace IS the narrative. Workflows execute against synthetic data AND real BoVerse client data (CSV-exported initially, live ServiceTitan MCP for the Apex case).

6. **The living co-design experience.** BoVerse staff sit with the SMB estimating manager. They drop in artifacts or press record on a voice memo. 90 seconds later the inferred summary appears on screen with confidence bars next to each rule. Hover over any rule to see the prompt, RAG asset, and reasoning. Click to correct conversationally.

## Approaches Considered

### Approach A: Lean Python stack (minimal viable)

Anthropic SDK direct + sequential Python + python-docx + pandoc. JSON state between stages. RAG via files in prompts.
Effort: M (human ~4 weeks / CC ~40 hours). Risk: Low.
Pros: smallest deps, easiest debug, pure Python. Cons: write all orchestration yourself, no native subagent isolation.

### Approach B: Claude Agent SDK orchestration (CHOSEN)

Claude Agent SDK with subagents per stage + 3 MCP servers (ingestion + state + ServiceTitan) + pgvector for RAG + python-docx + Jinja2 + pandoc + Streamlit for UI + Whisper for voice + Alembic for migrations + DeepEval for evals.
Effort: L (human ~26 weeks / CC ~170 hours baseline + risk-adjusted with expansions). Risk: Medium.
Pros: subagent isolation, MCP plugin architecture, prompt caching by SDK, scales beyond prototype.
Cons: more deps, learning Agent SDK while building.

### Approach C: n8n-as-runtime (lateral)

BoVerse builds generation layer only. n8n runs the workflow. 400+ integrations free.
Effort: M (human ~3 weeks / CC ~30 hours). Risk: Medium-High (n8n lock-in).

## Recommended Approach

**Approach B: Claude Agent SDK orchestration, with EXPANSION layer + locked architectural decisions D1-D7.**

### Stage-by-stage architecture

- **Stage 1 (Dump & Ingest).** `boverse-ingestion-mcp` server exposing 5 tools (PDF, email, spreadsheet, text, voice). Backwards-inference subagent (Sonnet, high reasoning) reads from `boverse-state-mcp` and writes inferred summary back.
- **Stage 2 (Targeted Clarification).** Gap-analysis subagent + question-generation subagent. Both read inferred summary from state MCP, write clarification questions/answers to state MCP.
- **Stage 3 (Simulation Pack).** Schema-design subagent + synthetic-data subagent. Read clarified inference from state, write schema + 10 rows back.
- **Stage 4 (Workflow Generation).** Five sub-subagents (primitive-mapper, actor-assignment, model-selection, prompt-chain-design, RAG-library-spec). Each reads relevant context from state MCP, writes workflow JSON fragments back.
- **Stage 5 (Output Bundle + Real Execution).** Word-doc-generation + demo-run + trace-narration subagents. Demo-run executes generated workflow against synthetic + real data (E1a CSV / E1b live MCP via ServiceTitan).

### Three MCP Servers

1. **`boverse-ingestion-mcp`** (local stdio). 5 tools: `ingest_pdf`, `ingest_email`, `ingest_spreadsheet`, `ingest_text`, `ingest_voice_note`. Per-tool deps: pdfplumber, mailparse, openpyxl, plaintext, Whisper API. Each tool writes extracted content directly into state via state MCP.

2. **`boverse-state-mcp`** (local stdio, third MCP, locked in D2). Typed DB tools for engagement state. Tools include: `create_engagement`, `save_extracted_content`, `get_extracted_content`, `save_inference`, `get_inference`, `save_clarification_answers`, `save_schema`, `save_synthetic_data`, `save_workflow_spec`, `get_workflow_spec`, `save_execution_trace`, `get_metrics`. All scoped by `engagement_id`. Uses psycopg3 + SQLAlchemy under the hood.

3. **`boverse-servicetitan-mcp`** (local stdio, mode-switchable). 3 tools: `query_quote_requests`, `query_client_history`, `query_labor_rates`. Config flag `BOVERSE_SERVICETITAN_MODE=mock|live` selects implementation at startup. Mock-MCP fallback returns canned data matching the schema if sandbox access denied.

### Subagent vs function call decision rule

Subagent if it requires LLM judgment. Function call if deterministic transformation. The 22-step Apex example splits roughly: ~14 subagents, ~8 function calls. Per-step assignment finalized during Stage 4 actor-assignment subagent's run.

### Workflow runtime scope

Deliverable 2 targets **5 happy-path + 2 edge-case-triggering steps** executing end-to-end. The other 15 Apex steps are stubbed with the same interface but mocked outputs. **For the April demo: be explicit with viewers that 7/22 steps are real execution and 15 are interface stubs** (outside voice finding #13 — presentational risk if not framed honestly).

### Backwards inference subagent contract

**Input:** array of extracted material objects from Stage 1 ingestion tools.

**Output JSON:**
```
{
  "inferred_business_type": "string",
  "inferred_process_name": "string",
  "inferred_output_goal": "string",
  "inferred_inputs": ["string"],
  "inferred_steps": [{"step": "string", "primitive": "ingest|transform|validate|action|feedback", "confidence": 0.0-1.0}],
  "inferred_rules": ["string"],
  "inferred_knowledge_assets": ["string"],
  "what_we_cannot_see": ["string"],
  "summary": "2-3 sentence plain-English"
}
```

**Prompt strategy:** Sonnet, high reasoning effort. System prompt establishes workflow-architect role with evidence-based reasoning constraint. User prompt injects extracted materials + industry context + stated output goal.

**Confidence scoring rubric** (injected into system prompt): 0.9+ = directly visible, 0.7-0.9 = strongly implied, 0.5-0.7 = plausible inference, <0.5 = guess. Items with confidence < 0.7 drive Stage 2 clarification.

### Iteration loop granularity (two surfaces, one backend)

Both Word doc and Streamlit chat feed the same spec-diff subagent. Word doc surface: pandoc → markdown → unified diff → spec-diff subagent. Chat surface: NL → spec-diff subagent. **Outside voice finding #5 caveat**: these two paths have hidden coupling. Word doc diff localizes change by line; chat input localizes nothing without inference. The spec-diff subagent does two different jobs. Plan budgets 1 extra week (week 11) for the chat-side prompt engineering this requires. Chat-side may need explicit "I think you mean X, confirm?" clarification turns (added as TODO).

### RAG

pgvector + Voyage AI free-tier embeddings (decision week 3). RAG assets chunked at section level. Cosine similarity retrieval.

### Document parsing

Default Unstructured.io. Week-2 bake-off against real Apex materials. LlamaParse fallback. **Pandoc round-trip risk #3 upgraded to HIGH likelihood** (outside voice finding #6). OOXML fallback via python-docx must be prototyped in week 2 alongside pandoc validation.

Single Streamlit multipage app, three panels, two user personas.

| Panel | User persona | Content |
|-------|--------------|---------|
| **Co-design** (E3) | SMB client + BoVerse operator together | Inferred summary, confidence bars, click-to-correct, hover annotations |

**Latency mitigation strategy** (per outside voice finding #4):
- Use `@st.cache_data` for read-only queries (RAG retrieval, prior inference)
- Selective regen: only re-run the affected sub-subagent, not all 5
- Async Anthropic SDK calls where the panel can show a spinner during regen

- Streamlit's full-script-rerun model is the framework default; if it becomes a hard blocker, consider extracting Co-design panel to FastAPI + minimal HTMX as v2.

**Auth boundary:** Analytics panel requires basic auth via env-var password. Decision week 6 on whether to graduate to Streamlit experimental auth.

### Error format (D11)

Every workflow exception inherits from `WorkflowError`. Primitive-level failures (ValidationError, TransformError, ActionError, FeedbackError) carry: `step`, `primitive`, `field`, `value`, `rule`, `source`. Rendering wraps the exception with a 4-block structure: **what failed** / **what it means** / **what to do (3 ranked options)** / **how to debug (specific commands)**. Stack traces go to file, not stdout (unless `-vvv`).

### Observability (D13)

Every generated workflow ships with:
- OTel auto-instrumentation (`opentelemetry-instrument` wrapper for `workflow run`)
- Spans per primitive with attributes (workflow_id, primitive_name, step, primitive_kind)
- Structured logs (structlog) with workflow_id + run_id + primitive_name
- Prometheus metrics endpoint (`/metrics` on port 9090) for primitive latency + success rate + error class
- Default Grafana dashboard JSON (`docs/operations/grafana-dashboard.json`)
- OTel collector container in `docker-compose.dev.yml` (prints traces to stdout)

### Upgrade path (D12)

Three independent semvers per generated artifact:
1. **Workflow version** (semver of the generated repo)
2. **Factory version** (semver of the BoVerse Factory that generated it, in `pyproject.toml`)
3. **Primitive version** (semver per primitive in `primitives/__manifest__.json`)

Breaking changes between regenerations trigger:
- Auto-generated Alembic migration (schema)
- Auto-generated state migration script (checkpoint format)
- Auto-generated config migration (env var renames)
- Rollback scripts for every above

`workflow diff <old-version> <new-version>` produces a human-readable diff showing primitive additions/modifications/removals, schema changes, state changes, model changes, and a recommended migration order.

## Evaluation Protocol (context-aware, DeepEval framework)

8 core dimensions (from rev 4) + 6 expansion-specific dimensions. **Framework: DeepEval** (locked in D4) with custom Python harness on top for ablation studies + calibration plots (DeepEval doesn't natively cover those).

### Core dimensions

1. Inference quality, vertical-adapted (per-vertical rubric from BoVerse domain expert)
2. Artifact complexity scoring (input difficulty grading curve)
3. Confidence calibration (correctness vs confidence bucket)
4. Ablation studies (full vs no-inference vs no-clarification vs no-RAG)
5. LLM-as-judge cross-validation (Opus vs domain expert)
6. Edge-case handling rate (3 of 3 caught at expected stage)
7. Iteration cycle quality (precision/recall/preservation per amendment, per surface)
8. Time-to-spec by stage

### Expansion-specific dimensions

9. Voice intake quality (Whisper WER ≤15% on Apex jargon). **Risk #17 caveat**: without domain adaptation, baseline WER is 20-30%. Mitigation: pass custom vocabulary prompt to Whisper.
10. Real-data execution quality (E1a CSV: ≥80% correctly priced or flagged)
11. Co-design UX latency (click-to-correct ≤5s, fallback ≤10s per outside voice finding #4)
12. Live MCP integration reliability (E4: ≥99% query success rate)
13. Analytics dashboard usefulness (BoVerse operator subjective rubric ≥7/10)
14. Chat iteration accuracy (same precision/recall as Word doc; surface comparison)

**Data confound caveat** (outside voice finding #12): for the inference quality measurement, distinguish "we inferred the workflow from artifacts" from "we executed a workflow that gets the right answer." Eval should separate these. Inference quality on rubric items (margin rules, multipliers) is independent of execution quality on quote outputs. Deliverable 3 reports both separately.

### Sample sizes + reporting

- 2 verticals × 1 representative workflow each = 2 workflows main protocol
- Iteration loop: 5 amendments per workflow per surface (doc + chat) = 20 cycles total
- Ablations: 3 extra runs on Apex
- LLM-as-judge: every output
- Calibration: ≥50 inferred items
